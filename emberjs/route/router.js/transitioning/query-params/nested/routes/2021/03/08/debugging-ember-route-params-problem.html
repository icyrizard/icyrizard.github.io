Debugging a 'Transition Aborted' in Ember

 
  • Mar 08, 2021

I was having a problem the other day when Ember start aborting a Transition after clicking on a link. Imagine a component that can trigger a redirection to some resource if it had the model, or model.id. For this you can either generate a link using the link-to helper:

     [1] will trigger the model hook with a params object containing the id of the model
     {{#link-to "view.resource" model.id}}Click to view this resource!{{/link-to}}

     [2] will trigger the model hook of the route, because we pass-in th model
     {{#link-to "view.resource" model}}Click to view this resource!{{/link-to}}

Another solution is to use this.transitionTo() function in a route. It allows for programmatically transitioning to other routes. In our code base we have a transitionTo action defined that allows triggering transitions from anywhere to anywhere as long as you can access the route action.

Notes on Ember (Nested) Route Transitions

When Ember starts to redirect to a different route, it will travers from the toplevel route to the child route. Meaning that if you go to “view.resource”, Ember will first transition to “view”, and then to “resource”. Those routes in the same “stack” of cards if you will, they belong together are only valid in that order.

Now imagine that the level of nested routes consists of one level more. So three layers, (e.g., “view.resource.details”). When traversing to the deepest route, any of the other two routes can generate problems and abort the transition for any given reason. Often they are aborted due to some failing promise, or in this case it was a bit more complicated.

See here the code that was executed without a propper stacktrace in the router.js library. This is the library Ember uses under the hood to transition between routes. It was a bit tricky without external help I must say.

  rollback() {
    if (!this.isAborted) {
      log(this.router, this.sequence, this.targetName + ': transition was aborted');
      if (DEBUG) {
        let error = new Error(`Transition aborted stack`);

        this.debugAbortStack = () => error.stack;
      }

      if (this.intent !== undefined && this.intent !== null) {
        this.intent.preTransitionState = this.router.state;
      }

      // IMPORTANT: [1] Remember this line here. This 'isAborted' is a property was changed!
      this.isAborted = true;
      this.isActive = false;
      this.router.activeTransition = undefined;
    }
}:

Listings 1: Code find in here in the router.js library: transition.ts

The above rollback() was triggered, but who triggered it, where? Which route? To answer this question I needed to dive deep in to the rabbit whole. But remember the isAborted, which is important becauce that is flipped to false, and I need to know why and where.

Steps to reproduce (recap)

My actions to trigger the problem were:

  1. Click on trigger the redirection, wait for it to trigger a redirect using transitionTo.
  2. Transitions from “other-view” Route with QueryParams ?resource_id=987 to view.resource.details modelId:123
  3. Ember aborts one of the routes and triggers a rollback of the transition, without giving a proper reason.

Now I noticed that with query params defined in the start route, it fails, but without them it does not… That started smelling like a cause of the problem.

Query Params causing problems

So as stated above, Ember uses route.js library for its routing. There I can find the code that sets the flag isAborted to false on the transition object.

  1. When transitioning to a nested route (which is simply a child of some route).
  2. It starts with the outer parent route and moves into the child. Based on the given parameters to the route it will trigger the model hook or not. If it’s an ID it will trigger the model hook with parameters, if you pass a model, it won’t trigger the model hook. Instead it will call the afterModel and all the things that come after the model hook.
  3. One of each transition in the tree of routings can fail, which one can be a bit difficult to find.
    • 3a. It crashed right here: https://github.com/tildeio/router.js#L173
  4. The transition was aborted for some reason, the variable isAborted of one of the Transitions in the stack was flipped to false… Still, no idea why yet.

Solution, watch ‘isAborted’ flip to False!

I’ve used a small snippet to called watch. It triggers when the variable is changed value. As noted before, the isAborted variable denoted in Listing 1 was set to false. You need to add the snippet, just paste into the console. This will change the behaviour of all JS objects. You will need to set a breakpoint before the variable is changed, then call the snippet:

And watch:

this.watch(‘isAborted’, function() {
  // trigger debugger
  debugger;
});

It will hit the debugger when the value changes from true to false, then, see the stack trace and debug. Then I found that the Transition was aborted because of a refresh that was triggered in a route that was not active. The reason for it was that the QueryParameters were changing, and the transitioning was taking place at the same time. Normally that is not a problem, but due to some old lingering code in one of the routes, this caused the problem.

Remember that the QueryParams were set to: ?resource_id=987, while the transition was being executed from the “other-view” route to the “view.resource.details”, the first thing that happened was that the query params changed. In Ember you can have a hook on the Rout called queryParamsDidChange. This was specifically done in a file route name listRoute, a common used code to show list resources. Within this hook a refresh is automatically triggered if the queryParaams change, that’s pretty usefeul! But this is triggered before the route was actually changed, it was actually the first thing that happened. Nothing was loaded yet, no models were set up, ‘the transition was still on the move.’. But this route was no longer active anymore. Triggering a refresh on a non-active route, can cause weird problems. Luckily you can test if the current route is active when the refresh is triggered, so finally, the solution. Behold;

queryParamsDidChange: function() {
-- this.refresh();
++  if (this.router.isActive(this.routeName)) {
++        this.refresh();
++ }
}
